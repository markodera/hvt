# HVT Tenant Isolation Architecture

This document details the architectural approach to user multi-tenancy and data isolation in the HVT Identity Provider platform.

## 1. The Core Problem: The Global Email Constraint
Before this architectural refactor, HVT was structured like a standard single-tenant Django application. The `User` model enforced email uniqueness globally:
`email = models.EmailField(unique=True)`

As a multi-tenant Identity Provider, HVT operates across two distinct planes:
* **Control Plane:** Developers who log into the HVT dashboard to manage their organizations.
* **Runtime Plane:** End-users (customers) who sign up for the apps built by the Control Plane developers.

Because of the global email constraint, user pools were bleeding into each other. If `john@example.com` signed up for Developer A's fitness app, HVT prevented `john@example.com` from signing up for Developer B's e-commerce app, throwing a "User with this email already exists" error.

## 2. The Solution: Two-Plane Isolation Architecture
To achieve true Auth0-level isolation, the system was refactored to treat the database as partitioned silos based on the `project_id` injected via the `X-API-Key`.

### A. Database Model Changes (`apps/users/models.py`)
We removed the global uniqueness of the email field and replaced it with strict, scope-aware constraints at the PostgreSQL level:

```python
email = models.EmailField(db_index=True)  # unique=True removed

class Meta:
    constraints = [
        # 1. Runtime Plane Isolation (End-users)
        # The same email can exist millions of times, as long as each 
        # belongs to a different project.
        models.UniqueConstraint(
            fields=["email", "project"],
            name="uniq_user_email_per_project",
        ),
        # 2. Control Plane Isolation (Developers)
        # Developers managing HVT must remain globally unique.
        models.UniqueConstraint(
            fields=["email"],
            condition=Q(project__isnull=True),
            name="uniq_user_email_when_project_null",
        ),
    ]
```

### B. Identity Resolution Layer (`apps/authentication/identity.py`)
Because the database now contains duplicate emails, writing `User.objects.get(email="john@example.com")` will instantly crash the system with a `MultipleObjectsReturned` error. 

To solve this, all user queries are now routed through a dedicated identity layer that enforces API Key context:

* **`get_control_plane_users_by_email`**: Retrieves dashboard developers by ensuring `project__isnull=True`.
* **`get_runtime_user_for_api_key(email, api_key)`**: Resolves the exact Runtime user by ensuring the user belongs to both the `organization_id` and `project_id` connected to the active API key.

By ensuring every authentication lookup requires the API key instance, it becomes mathematically impossible for an API request to interact with or authenticate a user from a different tenant.

### C. Serializer and Flow Separation
The legacy standard serializers (like `dj-rest-auth`'s `PasswordResetSerializer` or `LoginSerializer`) inherently perform global email lookups. These were completely replaced with plane-specific logic:

**Registration:**
Registration serializers no longer check global existence. `RuntimeRegisterSerializer` limits existence validation to the boundaries of the associated API key project. 

**Password Reset (`apps/authentication/serializers.py`):**
Password reset workflows split into:
1. `ControlPlanePasswordResetSerializer` -> Finds users where project is null.
2. `RuntimePasswordResetSerializer` -> Validates the email against the specific project attached to the active `X-API-Key`.

**Login & Signals (`apps/authentication/signals.py`):**
Auth hooks and signals (like login failure tracking) now inspect the `request.auth` object. If an API key is present, it looks for the user in the Runtime Plane; otherwise, it defaults to the Control Plane.

## 3. Backward Compatibility & Migrations
To ensure this did not break existing integrations:
1. **Migration Handlers:** Custom RunPython migrations were written to safely drop the strict `unique_verified_email` indexes generated by libraries like `django-allauth` without wiping data.
2. **Legacy User Fallbacks:** If a user was created in an older version of the system where they had an `organization_id` but lacked a `project_id` assignment, the new `user_matches_runtime_project` function gracefully allows them into their legacy structure while strictly enforcing the rules for newly created accounts.
